# 背景

描述需求提出的背景，或触发进行优化的问题。并简要说明本特性/优化想要达成的目标。

# 定义

文档中所使用的一些并非众所周知的术语的定义。

# 行为说明

如果有行为变化，对产品行为进行详细说明，涉及配置、发版方式、部署方式、SQL 命令等各个角度。对有的产品也可能会涉及到用户界面、编程接口（供用户的应用程序调用的接口）等的变化。总之，一切用户、发版者、部署者能够感知到的行为变化，均需进行详细说明。下面是细节要求：

- 对于涉及的 SQL，必须提供示例语句。涉及数据处理的 SQL 要使用现有文档的智能电表的示例数据来写示例。
- 对于 API，必须提供示例代码（示例代码要作为测试用例加入 CI）；要有返回的错误码列表，对每个错误码要做说明
- 如果引入新的参数或新的概念，要有定义和说明，配置参数指明有效范围和缺省值
- 出错处理，对各种异常情况进行说明
- 如果引入发版行为的变化

  - 对于 Docker，需提供 Dockerfile 或伪代码
  - 对于安装包，需提供详细的组件/文件依存关系、目录创建、文件复制、脚本运行等说明，以代码或伪代码形式提交

实际在写用户手册时，如果行为说明较复杂，为了避免章节嵌套过多，也可以将行为说明拆分到多个一级标题下。

如果没有行为变化，则本节内容为无。

# 性能

描述本需求对写入、查询、启动或其它可能的场景中带来的性能影响。如果没有则写无，并简要说明理由。如果为性能优化类，则在本节说明优化后的目标。

# 安全

描述本需求对安全的设计和考量，以确保数据的机密性、完整性和可用性。

# 兼容性

如果新的产品行为对旧有行为产生了破坏，要在本节明确说明，并阐述必须这么做的理由。如果没有则本节标无。

# 运维

如果本特性有其它方面的影响，比如用户的部署方式，交付团队的客户支持等，在本节进行说明。如果没有则本节内容为无。

# 使用场景

对本特性或优化可能被用到的使用场景，即 Use Case，进行详细说明。尽可能穷举所有 use case。

# 约束和限制

约束：使用本特性/优化的约束条件，如果没有的话写无

限制：本特性有哪些限制，在使用中应该尽量避免

# 常见错误和排查

对使用中可能遇到的错误提示以及如何排查故障进行说明。很多小型优化或功能不会有复杂的错误、排查也不困难，可以标无。但对于一些复杂的功能，容易用错、使用中容易出错的，要进行说明。

# 可观测性

对 taos shell, taos Explorer， TDinsight 等基于 UI 或用户交互界面的产品组件是否有影响，如果有则说明清楚这几个组件的行为变化

# 安装和卸载

描述对安装和卸载脚本有什么要求才能够使用户能够顺畅地使用本特性/优化
