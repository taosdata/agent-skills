## 威胁识别与分析（STRIDE）

功能描述。

### 关键实体与数据（示例）

| 实体/数据 | 描述 | 敏感性 |
| --- | --- | --- |
| 本地缓冲区文件 | 存储在客户端设备磁盘上的暂存数据文件。 | 高（含原始时序数据） |
| 缓冲区元数据 | 记录文件路径、大小、校验和、状态（待发送/已发送）。 | 中 |
| 批量上传协议 | 客户端与服务器之间用于传输批量数据的专用协议。 | 高 |

### 威胁评估（示例）

| 威胁 ID | 威胁描述 / 攻击场景 | STRIDE | 相关组件 / 数据流 | 风险等级 |
| --- | --- | --- | --- | --- |
| T-ASYNC-01 | 本地缓冲区数据泄露：攻击者或恶意软件可读取客户端设备上未加密的缓冲区文件，获取敏感的时序数据（如传感器读数、设备状态）。 | I（信息泄露） | 本地缓冲区文件 | 高 |
| T-ASYNC-02 | 缓冲区数据篡改：攻击者篡改缓冲区文件内容，例如修改电表读数、设备状态等，导致污染数据库中的历史数据。 | T（篡改） | 本地缓冲区文件 -> 批量上传 | 高 |
| T-ASYNC-03 | 批量数据重放攻击：攻击者截获一次合法的批量上传数据包，并重复发送给服务器，导致数据重复插入。 | T（篡改） | 批量上传协议 | 高 |
| T-ASYNC-04 | 恶意数据注入：攻击者伪造或利用有漏洞的客户端，向缓冲区写入海量垃圾数据或格式错误的数据，耗尽服务器端处理资源。 | D（拒绝服务）/ S（仿冒） | 客户端 SDK -> 缓冲区 | 中 |
| T-ASYNC-05 | 缓冲区耗尽攻击：恶意应用持续生成大量数据，故意断网，写满客户端磁盘缓冲区，导致功能失效或应用崩溃。 | D（拒绝服务） | 客户端 SDK、本地磁盘 | 中 |
| T-ASYNC-06 | 抵赖性风险：客户端声称数据已成功写入缓冲区，但因故障丢失，系统无法审计是客户端未写入还是服务器未接收。 | R（抵赖） | 整个数据流 | 中 |
| T-ASYNC-07 | 元数据信息泄露：错误信息中返回缓冲区文件路径等敏感系统信息。 | I（信息泄露） | 错误处理 | 低 |

## 安全需求与设计约束（示例）

| 威胁 ID | 转化后的安全需求 / 设计约束 | 类型 | 优先级 |
| --- | --- | --- | --- |
| T-ASYNC-01 | SEC-ASYNC-001（数据加密）：客户端缓冲区文件必须使用强加密算法（如 AES-256-GCM）进行加密，密钥由应用提供或与用户会话绑定，而非硬编码。 | 安全功能需求 | 高 |
| T-ASYNC-02 | SEC-ASYNC-002（数据完整性校验）：每个缓冲区数据块在写入文件前必须计算数字签名或 HMAC。服务器端在接收批量数据时，必须强制校验完整性签名，校验失败则丢弃整批数据。 | 安全设计约束 | 高 |
| T-ASYNC-03 | SEC-ASYNC-003（防重放）：批量上传协议必须包含时间戳和一次性随机数，服务器端应验证请求的新鲜性，防止重放。 | 安全设计约束 | 高 |
| T-ASYNC-04 | SEC-ASYNC-004（输入验证与速率限制）：服务器端的批量接入点必须对数据格式和内容进行严格校验。同时，应对单个客户端源的批量上传频率和数量进行限制。 | 非功能安全需求 | 中 |
| T-ASYNC-05 | SEC-ASYNC-005（缓冲区管理）：SDK 必须提供可配置的缓冲区大小上限和旧数据自动轮转清理策略，防止磁盘写满。 | 健壮性需求 | 中 |
| T-ASYNC-06 | SEC-ASYNC-006（增强审计）：SDK 应记录重要操作日志（如缓冲区创建、数据写入、上传开始 / 成功 / 失败）。服务器端需记录每个批量请求的客户端 ID、时间、数据量，实现双向溯源。 | 审计需求 | 中 |
