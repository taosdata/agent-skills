# TAOS Logging Style
## 何时使用
- 需要新增、重构或评审 TDengine 相关日志时。
- 需要跨模块/跨进程追踪请求链路时。
- 需要基于日志定位“成功/失败原因”或并发问题时。

## 核心目标
1. 日志服务于“定位问题的人”，不是只给写代码的人看。
2. 日志必须可读、可追踪、可定位、可过滤。
3. 在保证定位能力的前提下保持精简，避免明显资源浪费。

## 日志格式
统一格式：
`MM/DD HH:MM:SS.UUUUUU THREAD_ID MOD QID:0x<HEX> <完整句子>`

约束：
- 时间戳精确到微秒。
- `MOD` 使用三字母模块缩写。
- `QID` 为请求链路 ID，统一十六进制输出。
- 正文必须是完整句子，能够独立表达语义。
- 变量值优先打印可读字符串；若有数字码（如消息 ID、错误码），应同时输出对应字符串含义。

## 写日志的硬性要求
以下场景必须打日志：
1. 接收到模块/进程消息时（第一时间）。
2. 发送消息给模块/进程时。
3. 定时器超时时。
4. 用户输入或第三方接口调用时（包含关键输入，如完整 SQL）。
5. 资源分配/异步释放且存在泄露风险时。
6. 失败操作（必须包含失败对象、系统错误码、错误字符串）。
7. 影响关键判断分支的变量值。

每个任务日志应满足：
- 能看出任务流程（跨线程、跨模块、跨进程）。
- 能看出成功或失败及原因。
- 能区分并发下的不同任务与同一任务的多次交易。

## 禁止写的日志
1. 函数入口/出口日志（临时调试后必须移除）。
2. 顺序执行语句的中间结果刷屏日志。
3. 一个函数内部的循环语句不能有日志。

## QID 规范
- QID 是 8 字节 request ID，用于端到端关联一次上游操作（如插入、查询）。
- 链路上的关键日志都应携带 QID，确保可按 QID 汇总消息流。
- ID 未分配前使用 `0`；某字段无对应 ID 时也填 `0`。
- 建议按固定段组织，便于部分匹配过滤（链路问题可只过滤前缀段）。

参考组成（按规范文档）：
- taosAdapter ID：8 bits
- taosX ID：4 bits
- taosx-agent/链路相关 ID：12 bits
- data source ID：8 bits
- transaction ID：32 bits（自然递增，溢出回绕）

## 工程约束
- 新日志文件创建时，先打印软件版本和 Build 信息。
- 新日志文件创建时，打印关键配置信息。
- 不输出源码文件名/行号（避免日志膨胀、降低可读性）。
- 日志需支持级别（info/warn/debug 等）且可动态调整。
- 日志文件需有保留与轮转策略（多文件循环或按时间切分）。
- 默认目录 `/var/log/taos`，允许配置。

## 排障输出要求
当需要定位问题时：
1. 先根据用户基础信息（如 user/email/IP/时间）定位候选链路。
2. 再按 QID（或 QID 前缀）聚合日志。
3. 输出按时间排序的消息流，指出异常开始点、传播路径、最终失败点。

命令建议遵循当前环境偏好，优先使用 `rg` 做日志过滤（必要时兼容 `grep`）。

## 生成日志文案的写法模板
- 接收消息：`received <message_name> from <peer>, qid: <qid>, payload size: <n> bytes`
- 发送消息：`sent <message_name> to <peer>, qid: <qid>, payload size: <n> bytes`
- 成功结果：`<operation> succeeded, qid: <qid>, affected rows: <n>, elapsed: <ms> ms`
- 失败结果：`<operation> failed, qid: <qid>, target: <target>, errno: <code>(<errstr>)`

## 执行时检查清单
在产出任何日志方案或代码改动前，确认：
- 是否使用统一格式并带 QID。
- 是否覆盖“收/发/超时/输入/失败/关键分支”必打点。
- 是否避免入口出口与中间刷屏日志。
- 是否保证可读性（完整句子 + 可读字符串）。
- 是否满足可追踪并发任务与链路级定位。
