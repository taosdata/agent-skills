# Schema and Data Generation Reference

Complete reference for defining data schemas and generation methods.

## Schema Structure

```yaml
schema:
  name: table_name
  tbname:
    prefix: prefix
    count: number
    from: start_index
  columns:
    - name: column_name
      type: data_type
      # ... generation parameters
  tags:
    - name: tag_name
      type: data_type
      # ... generation parameters
  generation:
    interlace: 1
    rows_per_table: count
    rows_per_batch: size
    tables_reuse_data: true
    num_cached_batches: 0
```

## Table Naming

```yaml
tbname:
  prefix: d
  count: 10000
  from: 0
```

**Parameters:**
- `prefix`: Table name prefix (e.g., `d` → `d0000`, `d0001`, ...)
- `count`: Total number of tables to create
- `from`: Starting index for numbering

**Examples:**
- `prefix: sensor, count: 100, from: 1` → `sensor0001` to `sensor0100`
- `prefix: meter, count: 1000, from: 0` → `meter0000` to `meter0999`

## Data Types

**Integer types:**
- `bool` - Boolean (0 or 1)
- `tinyint` - 8-bit signed integer (-128 to 127)
- `tinyint unsigned` - 8-bit unsigned (0 to 255)
- `smallint` - 16-bit signed (-32768 to 32767)
- `smallint unsigned` - 16-bit unsigned (0 to 65535)
- `int` - 32-bit signed
- `int unsigned` - 32-bit unsigned
- `bigint` - 64-bit signed
- `bigint unsigned` - 64-bit unsigned
- `timestamp` - Timestamp (milliseconds, microseconds, or nanoseconds)

**Floating-point types:**
- `float` - 32-bit floating point
- `double` - 64-bit floating point

**String types:**
- `binary(N)` - Fixed-length binary string (N bytes)
- `varchar(N)` - Variable-length string (N bytes)
- `nchar(N)` - Unicode string (N characters)

**Note:** Currently unsupported types: `json`, `geometry`, `varbinary`, `decimal`, `blob`

## Column Definition

### Basic Structure

```yaml
columns:
  - name: column_name
    type: data_type
    count: 1  # Optional: create multiple consecutive columns
    props: "encode='simple8b', compress='lz4', level='medium'"  # Optional: column properties
    gen_type: random  # Optional: data generation method (random, order, expression)
    # ... generation method parameters (see below)
```

**Parameters:**

- **name**: Column name. If count > 1, name is used as prefix (e.g., `name: current`, `count: 3` creates current1, current2, current3).
- **type**: Data type (see Data Types section above)
- **count**: Number of consecutive columns of this type. Default: 1
- **props**: Column property info for TDengine compression, may include:
  - `encode`: First-level encoding algorithm
  - `compress`: Second-level compression algorithm
  - `level`: Compression level
- **gen_type**: Data generation method. Options:
  - `random`: Random generation (default)
  - `order`: Sequential natural number growth (integer only)
  - `expression`: Generated by Lua expression

### Generation Method 1: Random with Range

For numeric types, generate random values within a range.

```yaml
- name: temperature
  type: float
  gen_type: random
  distribution: uniform
  min: -20.0
  max: 50.0

- name: count
  type: int
  gen_type: random
  min: 0
  max: 1000

- name: voltage
  type: int unsigned
  gen_type: random
  min: 200
  max: 240
```

**Parameters:**
- `gen_type`: Must be set to `random` (or omit, as it's the default)
- `distribution`: Random distribution model. Currently only `"uniform"` is supported. Default: `"uniform"`
- `min`: Minimum value (generated value ≥ min)
- `max`: Maximum value (generated value < max)
- `values`: Alternative to min/max, provide a list of values to randomly select from

### Generation Method 2: Sequential Order (Integer Only)

Sequential natural number growth. Value wraps back to min after reaching max.

```yaml
- name: seq_id
  type: int
  gen_type: order
  min: 1
  max: 1000
```

**Parameters:**
- `gen_type`: Must be set to `order`
- `min`: Minimum value (starting value), generated value ≥ min
- `max`: Maximum value, generated value < max
- When value reaches max, it wraps back to min

### Generation Method 3: Random from Values

Choose randomly from a predefined list of values.

```yaml
- name: status
  type: int
  values: [0, 1, 2, 3]

- name: location
  type: binary(24)
  values:
    - Beijing
    - Shanghai
    - Shenzhen
    - Guangzhou

- name: category
  type: varchar(16)
  values:
    - TypeA
    - TypeB
    - TypeC
```

### Generation Method 4: Expression (Lua)

Use Lua expressions for calculated or pattern-based values. Supports integer, float, double, and string types. If gen_type is not explicitly specified but the expr attribute is present, gen_type will be automatically set to expression.

```yaml
- name: phase
  type: float
  gen_type: expression
  expr: _i * math.pi % 180

- name: sine_wave
  type: float
  expr: math.sin(_i * 0.1) * 100

- name: incremental
  type: int
  expr: _i * 10

- name: random_walk
  type: float
  expr: _last + math.random(-5, 5)
```

**Parameters:**
- `gen_type`: Set to `expression` (or omit if expr is present, it will be auto-detected)
- `expr`: Lua expression string

**Available variables:**
- `_i`: Current call index (starts at 0, increments with each row)
- `_table`: Current table identifier (numeric)
- `_last`: Previous value (numeric types only, useful for random walks)

**Available functions:**
- Math: `math.sin`, `math.cos`, `math.tan`, `math.asin`, `math.acos`, `math.atan`
- Math: `math.sqrt`, `math.pow`, `math.exp`, `math.log`, `math.abs`
- Math: `math.floor`, `math.ceil`, `math.min`, `math.max`
- Math: `math.random()`, `math.random(n)`, `math.random(m, n)`
- Constants: `math.pi`, `math.e`
- String: `string.format`, `string.len`, `string.sub`, `string.upper`, `string.lower`

**Expression examples:**
```lua
-- Sine wave pattern
math.sin(_i * 0.1) * 50 + 50

-- Sawtooth wave
(_i % 100) * 2

-- Random walk
_last + math.random(-10, 10)

-- Exponential growth
100 * math.exp(_i * 0.01)

-- Periodic with noise
50 + math.sin(_i * 0.05) * 20 + math.random(-5, 5)

-- Table-based variation
_table * 10 + math.random(0, 100)

-- Complex multi-factor expression (from official docs)
(math.sin(_i / 7) * math.cos(_i / 13) + 0.5 * (math.random(80, 120) / 100)) * ((_i % 50 < 25) and (1 + 0.3 * math.sin(_i / 3)) or 0.7) + 10 * (math.floor(_i / 100) % 2)
```

This complex expression combines various math functions, conditional logic, periodic behavior, and random noise:
- **Part A**: `math.sin(_i / 7) * math.cos(_i / 13)` - Dual-frequency modulation (base signal)
- **Part B**: `0.5 * (math.random(80, 120) / 100)` - Adds 80%~120% random noise
- **Part C**: `((_i % 50 < 25) and (1 + 0.3 * math.sin(_i / 3)) or 0.7)` - Switches gain every 50 calls
- **Part D**: `10 * (math.floor(_i / 100) % 2)` - Switches baseline every 100 calls (0 or 10)

### Timestamp Column

Timestamp columns have special parameters.

```yaml
- name: ts
  type: timestamp
  start: now
  precision: ms
  step: 1
```

**Parameters:**
- `start`: Start time
  - `now`: Current time
  - `now + 10s`: Current time plus offset
  - `now - 1h`: Current time minus offset
  - `2024-01-01 00:00:00`: Absolute time
  - Offsets: `ns` (nanoseconds), `us` (microseconds), `ms` (milliseconds), `s` (seconds), `m` (minutes), `h` (hours), `d` (days)
- `precision`: Timestamp precision
  - `ms`: Milliseconds
  - `us`: Microseconds
  - `ns`: Nanoseconds
- `step`: Increment per row (in the specified precision unit)

**Examples:**
```yaml
# Start now, increment by 1ms
- name: ts
  type: timestamp
  start: now
  precision: ms
  step: 1

# Start 1 hour ago, increment by 1 second (1000ms)
- name: ts
  type: timestamp
  start: now - 1h
  precision: ms
  step: 1000

# Start at specific time, microsecond precision
- name: ts
  type: timestamp
  start: 2024-01-01 00:00:00
  precision: us
  step: 1
```

## Tag Definition

Tags are immutable metadata in TDengine. They use the same generation methods as columns.

```yaml
tags:
  - name: groupid
    type: int
    min: 1
    max: 10

  - name: location
    type: binary(24)
    values:
      - Region-A
      - Region-B
      - Region-C

  - name: device_type
    type: varchar(32)
    values:
      - Sensor
      - Actuator
      - Controller
```

**Key differences from columns:**
- Tags are set once per table (immutable)
- Tags are used for table grouping and filtering
- Tags don't change over time

## Generation Settings

```yaml
generation:
  interlace: 1
  rows_per_table: 1000
  rows_per_batch: 10000
  tables_reuse_data: false
  num_cached_batches: 0
```

**Parameters:**

- **interlace**: Interleaved write pattern
  - `1`: Round-robin (write 1 row to table1, 1 row to table2, ...)
  - `10`: Write 10 rows to table1, then 10 to table2, ...
  - Higher values: Better cache locality, less realistic for time-series

- **rows_per_table**: Total rows per table
  - Positive number: Fixed row count
  - `-1`: Unlimited (for CSV import or continuous generation)

- **rows_per_batch**: Maximum rows per batch request
  - Typical values: 1000-50000
  - Higher: Better throughput, more memory
  - Lower: Lower latency, less memory

- **tables_reuse_data**: Whether all tables share the same data
  - `true`: All tables get identical data (faster, less memory, default)
  - `false`: Generate unique data for each table

- **num_cached_batches**: Number of batches to pre-generate
  - `0`: Generate on-demand (default, lower memory)
  - Positive: Pre-generate batches (faster, higher memory)

## CSV Data Source

Import data from CSV files instead of generating it.

```yaml
schema:
  name: meters
  from_csv:
    tags:
      file_path: /path/to/tags.csv
      tbname_index: 0
      exclude_indices: "1, 2"
    columns:
      file_path: /path/to/data.csv
      tbname_index: 0
      timestamp_index: 1
      timestamp_precision: ms
      timestamp_offset:
        offset_type: relative
        value: +10s
      repeat_read: false
  columns:
    - name: ts
      type: timestamp
      precision: ms
    - name: current
      type: float
    - name: voltage
      type: int
    - name: phase
      type: float
  tags:
    - name: groupid
      type: int
    - name: location
      type: binary(24)
  generation:
    interlace: 1
    rows_per_table: -1
    rows_per_batch: 10000
```

### CSV Tags Configuration

```yaml
tags:
  file_path: /path/to/tags.csv
  has_header: true
  tbname_index: 0
  exclude_indices: "1, 2"
```

**Parameters:**
- `file_path`: Path to tags CSV file
- `has_header`: Whether the file contains a header row. Default: `true`
- `tbname_index`: Column index for table name (0-based). Default: `-1` (inactive)
- `exclude_indices`: Column indices to exclude (comma-separated string). Default: empty (no exclusion)

**Example tags.csv:**
```csv
table_name,skip_col1,skip_col2,groupid,location
meter0001,x,y,1,Beijing
meter0002,x,y,2,Shanghai
meter0003,x,y,1,Shenzhen
```

With `tbname_index: 0` and `exclude_indices: [1, 2]`, this reads:
- Table name from column 0
- Skip columns 1 and 2
- Read groupid from column 3
- Read location from column 4

### CSV Columns Configuration

```yaml
columns:
  file_path: /path/to/data.csv
  has_header: true
  tbname_index: 0
  timestamp_index: 1
  timestamp_precision: ms
  timestamp_offset:
    offset_type: relative
    value: +10s
  repeat_read: false
```

**Parameters:**
- `file_path`: Path to data CSV file
- `has_header`: Whether the file contains a header row. Default: `true`
- `tbname_index`: Column index for child table name (0-based). Default: `-1` (inactive)
- `timestamp_index`: Column index for timestamp (0-based). Default: `-1` (inactive)
- `timestamp_precision`: Timestamp precision in CSV ('s', 'ms', 'us', 'ns')
- `timestamp_offset`: Adjust timestamps (optional)
  - `offset_type`: 'relative' (offset from now) or 'absolute' (fixed time)
  - `value`: Offset value (e.g., '+10s', '-1h', '2024-01-01 00:00:00')
- `repeat_read`: Whether to read data repeatedly when exhausted. Default: `false`

**Example data.csv:**
```csv
table_name,timestamp,current,voltage,phase
meter0001,1704067200000,45.3,220,1.57
meter0001,1704067201000,46.1,221,1.58
meter0002,1704067200000,44.8,219,1.56
```

### Timestamp Offset Examples

**Relative offset (from current time):**
```yaml
timestamp_offset:
  offset_type: relative
  value: +10s  # CSV timestamps + 10 seconds from now
```

**Absolute time:**
```yaml
timestamp_offset:
  offset_type: absolute
  value: 2024-01-01 00:00:00  # All timestamps relative to this time
```

**Offset formats for relative type:**
- Format: `±[value][unit]` (e.g., "+1d3h" means add 1 day 3 hours)
- Units: `y` (year), `m` (month), `d` (day), `h` (hour), `s` (second)
- Examples:
  - `+10s`, `-30s`: Seconds
  - `+2h`, `-6h`: Hours
  - `+1d`, `-7d`: Days
  - `+1d3h`: 1 day and 3 hours

## Best Practices

**Data generation:**
- Use expressions for realistic patterns (sine waves, random walks)
- Add random noise to expressions for natural variation
- Use `_last` for random walk patterns
- Consider `tables_reuse_data: false` for unique data per table (more realistic)
- Use `gen_type: order` for sequential integer values that wrap around

**Timestamp configuration:**
- Match precision to your use case (ms for most, us/ns for high-frequency)
- Use `start: now` for real-time simulation
- Use `start: now - Xh` for historical data
- Set `step` based on data frequency (1000 = 1 second for ms precision)

**Performance:**
- Larger `rows_per_batch`: Better throughput
- `interlace: 1`: More realistic time-series pattern
- `num_cached_batches: 0`: Lower memory usage
- `tables_reuse_data: false`: Unique data per table

**CSV import:**
- Set `rows_per_table: -1` to read all CSV data
- Use `timestamp_offset` to adjust old timestamps
- Use `repeat_read: true` for continuous testing
- Ensure CSV column order matches schema definition
